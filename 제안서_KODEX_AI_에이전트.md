# 2. 제안 개요

## (1) 제안 배경 및 목적

### 배경
삼성자산운용은 국내 ETF 시장 점유율 1위의 자산운용사로서, 130조원 이상의 ETF 시장에서 투자자에게 차별화된 서비스를 제공해야 하는 과제를 안고 있습니다.

현재 ETF 투자 정보는 다음과 같은 한계가 존재합니다:
- **정보 분산**: 수익률, 투자설명서, 뉴스 등이 각각 다른 시스템에 산재
- **실시간성 부족**: 시장 데이터와 상품 정보의 동기화 지연
- **개인화 부재**: 투자자 성향에 맞춘 맞춤형 정보 제공 미흡
- **확장성 한계**: 증가하는 고객 문의에 대한 인적 대응 한계

### 목적
본 프로젝트는 ETF 관련 정형 데이터(수익률, 시세 등)와 비정형 데이터(투자설명서, 뉴스 등)를 유기적으로 결합하여, **정확성·최신성·검증 가능성**을 갖춘 AI 챗봇 시스템을 구축합니다.

핵심 목표:
1. **에이전틱 RAG AI 챗봇**: 단순 질의응답을 넘어 자율적으로 정보를 탐색·분석·종합하는 AI 에이전트
2. **MCP 기반 데이터 파이프라인**: Model Context Protocol을 통해 사내 레거시 시스템과 AI 에이전트를 표준 프로토콜로 연결
3. **금융권 수준의 보안**: 데이터 암호화, 개인정보 마스킹, 가드레일을 통한 컴플라이언스 준수


### 핵심 기술 요소

**1. MCP(Model Context Protocol) 기반 데이터 연동**

MCP는 Anthropic이 제정한 오픈 표준 프로토콜로, AI 모델이 외부 데이터 소스 및 도구에 접근하기 위한 통일된 인터페이스를 제공합니다. 본 프로젝트에서는 MCP 표준을 준수하여 삼성자산운용 사내 레거시 시스템과 AI 에이전트를 연결합니다.

**MCP 표준 프로토콜 준수 방안:**
- 통신 규격: JSON-RPC 2.0 기반 요청/응답, SSE(Server-Sent Events) 기반 스트리밍
- MCP 3대 기본 요소 구현: Tools(도구 호출), Resources(데이터 조회), Prompts(프롬프트 템플릿)
- 각 MCP 서버는 `initialize` → `tools/list` → `tools/call` 표준 핸드셰이크 준수
- Tool 스키마: JSON Schema 기반 입력 파라미터 정의, 타입 검증 자동 수행

**사내 레거시 시스템 연동 아키텍처:**

```
[사내 레거시 시스템]                    [MCP 서버 레이어]

시세 API (REST)  ──→ [Adapter] ──→  etf-performance MCP Server
상품 DB (Oracle)  ──→ [Adapter] ──→  etf-products MCP Server
문서 관리 (NAS)   ──→ [Adapter] ──→  document-search MCP Server
뉴스 API (B2B)   ──→ [Adapter] ──→  news-search MCP Server
                                        ↓
                                   [MCP Client (AI 에이전트)]
                                        ↓
                                   [LLM (Claude)]
```

- **어댑터 패턴**: 각 레거시 시스템의 데이터 형식(SOAP, REST, JDBC 등)을 MCP 표준 Tool 스키마로 변환하는 어댑터 레이어 구축
- **데이터 변환 레이어**: 사내 DB 스키마(Oracle/MSSQL)의 데이터를 MCP Resource 형식으로 정규화. 종목코드, 날짜 형식, 통화 단위 등 표준화 처리
- **연결 보안**: VPN/Direct Connect를 통한 사내망 접근, 각 MCP 서버별 API 키 인증, 요청률 제한(Rate Limiting)
- **장애 격리**: 개별 MCP 서버가 독립 컨테이너로 운영되어, 특정 레거시 시스템 장애 시 다른 MCP 서버에 영향 없음

**MCP 서버 7개 구축:**
- 향후 신규 데이터 소스 추가 시 MCP 서버만 추가하면 되는 모듈형 확장 구조

**2. 멀티 에이전트 오케스트레이션**
- 라우터 에이전트: 사용자 의도를 분석하여 최적의 전문 에이전트로 라우팅한다
- 상담 에이전트: ETF 기본 정보 안내, 용어 설명, 투자 가이드를 제공한다
- 분석 에이전트: 수익률 분석, 상품 비교, 시장 동향 분석, 차트를 생성한다
- 추천 에이전트: 투자 성향별 맞춤 ETF 추천, 포트폴리오 구성을 제안한다
- 각 에이전트는 독립적인 시스템 프롬프트와 도구셋을 보유한다
- 에이전트 간 협업 워크플로우: 복잡한 질의 시 라우터가 2개 이상의 에이전트를 순차/병렬로 호출하여 결과를 종합한다 (예: 분석 에이전트의 수익률 데이터를 추천 에이전트가 활용)

**3. 슬롯 필링(Slot Filling) 및 자율 추론**
- 슬롯 필링: 사용자의 모호한 질의에서 API 호출에 필수적인 엔티티(종목명, 기간, 위험성향 등)를 능동적으로 추출한다. 미확보 정보가 있을 경우 사용자에게 추가 질문을 생성하여 필수 슬롯을 채운 후 검색/분석을 수행한다
  - 예시: "ETF 추천해줘" → 투자목적·위험성향·투자기간 슬롯 미충족 → "투자 목적(배당/성장/안정)과 선호하시는 투자 기간을 알려주세요" 자동 생성
- 자율적 다단계 추론: 사용자의 복잡한 질의를 하위 작업으로 분해하고, 필요한 정보를 탐색·수집·종합하는 과정을 자율적으로 수행하는 에이전트 아키텍처를 구현한다
- Next Best Action: 조회 결과 및 대화 맥락에 기반하여 연관 정보나 심화 분석 질문을 선제적으로 제안하는 동적 가이드 기능을 제공한다 (예: KODEX 200 수익률 조회 후 → "유사한 대형주 ETF와 비교하시겠습니까?" / "최근 관련 뉴스를 확인하시겠습니까?")

**4. RAG(검색 증강 생성) 파이프라인**
- 지능형 문서 분할(Chunking): 투자설명서, 운용보고서의 구조를 이해하는 시맨틱 청킹을 적용한다
- 메타데이터 통합 인덱싱: 종목코드, 문서 유형, 작성일 등 메타데이터와 함께 인덱싱한다
- 하이브리드 검색: 벡터 유사도 검색 + BM25 키워드 검색 + RRF(Reciprocal Rank Fusion)를 적용하여 검색 정확도를 확보한다
- 자가 교정(Self-Correction): 검색된 정보가 부적절하거나 부족할 경우 스스로 판단하여 검색 질의를 재작성(Query Rewriting)하거나 추가 검색을 수행하는 메커니즘을 포함한다
- Re-ranking: 초기 검색 결과에 대해 Cross-Encoder 기반 Re-ranking을 수행하여 상위 문서의 관련성을 정밀하게 재평가한다

**5. 금융 특화 가드레일**
- 금칙어 필터링: 욕설, 혐오 표현, 부적절 투자 권유를 차단한다
- 할루시네이션 체크: 답변 내 수치 데이터를 MCP 서버 조회 원본 데이터와 교차 검증하며, 근거 없는 수치가 포함된 경우 해당 문장을 자동 제거하거나 경고 표시를 삽입한다
- 컴플라이언스: "투자 권유가 아닙니다" 면책 조항을 자동 삽입하며, 금융소비자보호법을 준수한다
- 근거 표시: 답변에 사용된 문서 출처(문서명, 섹션명, 작성일)를 명시하여 검증 가능성을 확보한다

**5. 데이터 실시간 동기화**
- ETL 파이프라인: 시세 데이터 5분 간격 동기화, 일별 수익률 자동 갱신
- 문서 자동 갱신: 신규 투자설명서/운용보고서 업로드 시 자동 청킹 → 임베딩 → 벡터DB 적재
- 뉴스 API 연동: 금융 뉴스 API B2B 구독을 통한 실시간 뉴스 수집 및 인덱싱

**6. 서비스 제공 방식 (FunETF 웹뷰 연동)**
- 본 챗봇은 독립 서비스로 구축하며, 삼성자산운용 FunETF 앱에 WebView(iframe)로 제공
- iframe 임베딩 시 postMessage 기반 양방향 통신 인터페이스 구현
- 사용자 접근 시 JWT 토큰 기반 권한 관리 적용 (비로그인 사용자 제한 기능 포함)
- CORS 정책 및 CSP(Content Security Policy) 설정으로 허용된 도메인에서만 iframe 삽입 가능
- 반응형 레이아웃으로 FunETF 앱 내 다양한 화면 크기에 자동 대응

**7. 관리자 어드민 페이지 (이원화 제안)**

본 프로젝트의 핵심 목적인 AI 챗봇 응답 품질에 집중하기 위해, 관리자 기능을 최소안과 확장안으로 분리하여 제안합니다.

**8. RAG 솔루션 라이선스 구조**
- 당사는 자체 개발한 RAG 파이프라인 솔루션을 보유하고 있으며, 본 프로젝트에 활용
- 라이선스 형태: 프로젝트 전용 영구 라이선스 (삼성자산운용 귀속)
- IP 귀속 조건: 본 프로젝트에서 개발되는 MCP 서버 소스코드, 프롬프트, 학습된 모델 등 전체 지식재산권은 삼성자산운용에 귀속
- 제3자 보안 평가: 당사 RAG 솔루션에 대한 제3자 보안 수준 평가를 수행하며, 평가 결과를 프로젝트 착수 시점에 삼성자산운용에 제출한다
- 라이선스 비용은 견적서에 별도 명시한다

**솔루션을 포함하여 제안하는 사유:**
LLM API, 뉴스 데이터 API, RAG 솔루션은 본 시스템의 핵심 엔진으로서, 모델 선정·프롬프트 최적화·데이터 품질 관리가 시스템 개발과 일체화되어야 최적의 성능을 달성합니다. 솔루션을 개발비와 분리하여 별도 견적으로 제시하며, 상세 비용은 견적서에 명시합니다.

### MCP 서버 설계

MCP(Model Context Protocol)는 AI 모델과 외부 데이터 소스를 연결하는 표준 프로토콜입니다. 본 프로젝트에서는 다음 7개의 MCP 서버를 구축합니다:

| MCP 서버 | 기능 | 데이터 소스 | 응답 시간 목표 |
|----------|------|-----------|-------------|
| etf-products | ETF 상품 검색 및 목록 조회 | RDS Aurora | < 200ms |
| etf-detail | ETF 상세 정보 (구성종목, 보수 등) | RDS Aurora | < 300ms |
| etf-performance | 수익률 데이터 및 가격 히스토리 | RDS Aurora + Redis 캐시 | < 200ms |
| etf-compare | 다중 ETF 비교 분석 | RDS Aurora | < 500ms |
| document-search | 투자설명서/운용보고서 RAG 검색 | pgvector + OpenSearch | < 800ms |
| news-search | 금융 뉴스 검색 | 뉴스 API + OpenSearch | < 500ms |
| etf-recommend | 투자 성향별 ETF 추천 | RDS Aurora + LLM | < 1000ms |

### 벡터 DB 구축

**임베딩 모델**: OpenAI text-embedding-3-large (3072차원)
- 한국어 금융 용어에 대한 우수한 성능
- 종목명, 지수명 등 전문 용어 인식률 95% 이상

**청킹 전략**:
- Semantic Chunking: 문서 구조(목차, 섹션)를 인식하여 의미 단위로 분할
- Chunk Size: 500~1000 토큰, Overlap: 100 토큰
- 메타데이터 보존: 종목코드, 문서유형, 작성일, 섹션명

**인덱싱 구조**:
```
documents 테이블:
├── id (UUID)
├── content (TEXT) — 청크 원문
├── embedding (vector(3072)) — 임베딩 벡터
├── metadata (JSONB) — 종목코드, 문서유형, 작성일
├── document_id (UUID) — 원본 문서 참조
└── created_at (TIMESTAMP)

인덱스:
├── HNSW 인덱스 (벡터 유사도 검색, ef_construction=200)
├── GIN 인덱스 (메타데이터 JSONB 검색)
└── B-tree 인덱스 (document_id, created_at)
```

   - 프로덕션 환경은 VPC 내부에서만 접근, 외부 접근 차단
   - API 키, DB 인증 정보는 AWS Secrets Manager로 관리

### 테스트 방안 (품질 요건 QUR-1~4 대응)

**QUR-1: 테스트 수행**

- 인수 테스트: 삼성자산운용 담당자 참여, 실제 업무 시나리오 기반 검증

- 오류 데이터 포함 테스트: 잘못된 종목코드, 비정상 입력값 등 엣지케이스 검증

**QUR-2: 품질관리 방안수립**
- 분석/설계/개발/테스트 각 단계별 품질 체크리스트 제시

### 기술이전 계획

**MOP(Method of Procedure) 작업절차서:**
- 시스템 오픈 절차: 사전 점검 체크리스트 → 배포 승인 → 블루/그린 배포 → 헬스체크 → 트래픽 전환 → 정상 확인 → 완료 보고
- 장애 대응 절차: 장애 감지(CloudWatch 알람) → 1차 판단(30분 이내) → 심각도 분류(P1~P4) → 담당자 호출 → 원인 분석 → 조치 → 재발 방지 보고
- 정기 점검 절차: 월간 보안 패치, 분기 성능 최적화, 반기 LLM 모델 평가 절차를 상세 기술한다
- 롤백 절차: 배포 후 이상 발생 시 이전 버전으로 즉시 롤백하는 단계별 절차를 포함한다

**운영자 교육 (2일, 총 16시간):**

| 일차 | 시간 | 주제 | 내용 |
|------|------|------|------|

| Day 1 오후 | 4h | 문서 적재 및 벡터DB 관리 | 신규 투자설명서/운용보고서 업로드 → 자동 청킹 → 임베딩 → 벡터DB 적재 절차, 문서 삭제/갱신 방법 |

**MCP 레이어 RBAC(역할 기반 접근 제어) 설계:**

| 역할 | MCP 도구 접근 범위 | 데이터 접근 범위 | 관리 권한 |
|------|----------------|-------------|---------|
| 관리자(Admin) | 전체 7개 도구 | 전체 데이터 + 감사 로그 | 도구 추가/삭제, 프롬프트 수정, 사용자 관리 |
| 운영자(Operator) | 전체 7개 도구 | 운영 데이터 + 대화 이력 | 문서 적재, 프롬프트 수정, 모니터링 |
| 뷰어(Viewer) | 조회 도구만 (검색/상세) | 읽기 전용 | 대시보드 조회만 |
| AI 에이전트(System) | 할당된 도구만 | 도구 응답 데이터만 | 권한 없음 |

- 모든 MCP 도구 호출은 감사 로그(Audit Log)에 자동 기록하며, 호출자 ID, 도구명, 입력 파라미터, 응답 시간, 결과 상태를 포함한다
- 감사 로그 보존 기간: 최소 1년, CloudWatch Logs + S3 아카이브로 관리한다
- 관리자 대시보드에서 감사 로그 검색/필터링/다운로드 기능을 제공한다

**기술 문서:** 아키텍처 설계서, API 명세서, 운영 가이드를 전달한다

**인수인계 기간:** 2주 (마지막 안정화 기간에 병행 수행한다)

### 2. 데모 시스템 운영 현황
- 본 제안 시점에 이미 **동작하는 PoC 시스템**을 구축하여 운영 중
- 챗봇 UI + MCP 기반 도구 호출 + 멀티 에이전트 + 관리자 대시보드 동작 확인
- 실제 KODEX ETF 230개 상품 데이터 + 투자설명서 4건 기반 테스트 완료

### 3. LLM 모델 전략
- 메인 모델: Claude (Anthropic) — 한국어 성능 우수, 금융 도메인 정확도 높음
- 보조 모델: 경량 모델을 의도 분류, 슬롯 필링 등 단순 작업에 활용하여 비용 최적화
- 향후 삼성자산운용 자체 파인튜닝 모델을 적용할 수 있도록 모듈형 구조로 설계한다

### 4. 라이선스 및 3rd Party 솔루션 관리
- 라이선스 이슈, 비인가 S/W 설치 등으로 인해 발생되는 문제의 책임은 당사에 있다
- 본 프로젝트에 사용되는 모든 오픈소스 라이선스를 SBOM(Software Bill of Materials)으로 관리한다
- 3rd Party 솔루션(LLM API, 뉴스 API 등) 연동 시, 보안 취약점 발생에 대한 대응 체계를 갖추며 최종 책임은 당사가 부담한다

### 5. 프로젝트 범위 외 사항 (Out-of-Scope)

본 프로젝트의 범위에 포함되지 않는 사항을 아래와 같이 명시한다. 아래 사항에 대한 추가 요구 발생 시, 별도 협의 및 변경관리 절차를 통해 범위를 조정한다.

| 구분 | Out-of-Scope 항목 | 사유 |
|------|-----------------|------|
| 레거시 시스템 | 사내 기존 시세 시스템, 문서 관리 시스템, DB 등의 수정/개선 | 당사 시스템은 MCP 어댑터를 통해 연동하며, 원본 시스템 변경은 삼성자산운용 내부 영역에 해당 |
| FunETF 앱 | FunETF 모바일 앱 자체의 개발/수정/배포 | 본 챗봇은 iframe(WebView) 방식으로 삽입하며, 앱 자체 수정은 포함하지 않음 |
| 법률 자문 | 금융소비자보호법, 개인정보보호법 등에 대한 법률 자문 | 당사는 기술적 컴플라이언스 가드레일을 구현하며, 법률 해석 및 자문은 별도 전문기관 영역 |
| 사용자 개인정보 이관 | 기존 고객 개인정보(계좌, 투자내역 등)의 클라우드 이관 | 본 챗봇은 개인정보를 저장하지 않는 구조로 설계하며, 기존 개인정보 이관은 범위 외 |
| 하드웨어 조달 | GPU 서버, 네트워크 장비 등 물리적 하드웨어 구매/설치 | 클라우드 기반으로 구축하며, 온프레미스 하드웨어는 포함하지 않음 |
